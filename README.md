# Homework
简单讲讲，项目里有哪些模块、每个模块做了什么、还有大概是怎么串起来的。

## 1. 流程和场景切换

**做了什么：**  
主要把游戏流程跑通了：启动 -> 主菜单 -> 基地 -> 关卡战斗 -> 结算 -> 回基地。

**怎么设计：**  
用`GameStateMachineService`管状态，用`GameFlowOrchestrator`管“该进哪个场景”和“什么时候切换”。

**链路：**  
Boot里先装全局配置 -> 加载Persistent场景 -> 进主菜单 -> 选功能后进基地或关卡。

---

## 2. 主菜单和存档槽位

**做了什么：**  
主菜单点“开始游戏”后，先弹一个“3个存档槽位（左中右）”的窗口（E胖是对的）。  
下面再点“开始游戏”才会进入对应存档。

**怎么设计：**  
`MainMenuUI`负责界面和按钮交互，`SaveSlotService`负责当前选中的槽位和每个槽位的摘要信息。

**链路：**  
点Start -> 弹槽位窗口 -> 选槽位 -> 点Start Game-> 激活该槽位 -> 进入Base。

---

## 3. 存档和进度系统

**做了什么：**  
每个槽位各自有一份进度，记录已解锁关卡，已通关关卡和最后选择的关卡。（其实还有个成就系统，不过没加具体的成就）

**怎么设计：**  
`ProgressService`做读写存档，`ProgressSaveData`是存档数据结构，`SaveSlotService`负责“当前用哪个槽位文件”。

**链路：**  
选中槽位 -> `ProgressService`加载该槽位的json -> 游戏里发生进度变化就立即保存。

---

## 4. 关卡解锁规则

**做了什么：**  
解锁改成按通关推进，初始只有`body_1`  ，通关`body_1`解锁`body_2`  ，通关`body_2`解锁`soul_1`和`memory_1`（但是这两关没怎么设计,纯占位的）

**怎么设计：**  
在 `ProgressService.MarkRunCompleted(...)` 里统一做“记录通关 + 检查前置 + 自动解锁”。

**链路：**  
战斗胜利 -> 先标记当前 run 已通关 -> 自动检查并解锁后续关卡 -> 再进胜利结算页面。

---

## 5. 基地界面（选关 / 解锁窗口）

**做了什么：**  
基地里可以选关进入战斗；“unlock”窗口现在主要是展示每关是否已解锁，以及通关了那些关卡。

**怎么设计：**  
`BaseSceneUIController` 负责窗口和按钮，`ProgressService` 提供“解锁状态”和“通关状态”数据。

**链路：**  
玩家靠近交互点 -> 打开面板 -> 看状态/选关 -> 进入对应关卡。

---

## 6. 战斗玩法

**做了什么：**  
主要实现了移动、跳跃、射击、瞄准状态的摄像头和射击逻辑、换弹、切枪、技能释放（做了个简单的推开技能，敌人也使用的同样的技能系统）。

**怎么设计：**  
`PlayerController` 处理移动，`PlayerCombat` 处理战斗输入和武器切换，`WeaponBase`处理武器本体逻辑，`PlayerSkillSystem` 处理技能。

**链路：**  
输入采集 -> 玩家状态更新 -> 武器开火/命中 -> 事件广播 -> UI/音频响应。

---

## 7. 敌人和波次

**做了什么：**  
有普通敌人、远程敌人和Boss，支持波次刷怪、追击、攻击、死亡（死亡的时候还做了个简单的溶解的shader）、掉落。

**怎么设计：**  
`EnemyAIController`+`EnemyFSM`状态机控制行为，`EnemyCombat` 控制攻击，`LevelEncounterController` 控制整场战斗的刷怪和收尾。

**链路：**  
进入关卡 -> 初始化波次 -> 按配置刷怪 -> 敌人AI执行 -> 敌人死亡触发掉落和事件。

---

## 8. UI

**做了什么：**  
战斗里有玩家血条（数字在中间）、敌人血条、准心、当前武器名、弹药、技能冷却；战斗后有胜利/失败结算页。

**怎么设计：**  
主要是`PlayerHUD`、`DynamicCrosshair`、`GameResultUIController` 配合UI Toolkit。

**链路：**  
进入战斗时绑定玩家引用 -> 每帧刷新数值和状态 -> 结束时切到结果界面。

---

## 9. 音频和事件系统

**做了什么：**  
实现了基础BGM、UI点击音、战斗音效（开火、受击、敌人攻击/死亡等）。

**怎么设计：**  
`EventBus`做模块间广播，`AudioRuntimeService`订阅关键事件后播放对应音效。

**链路：**  
战斗/系统发事件 -> 音频服务收到事件 -> 播放匹配的音乐。

---

## 10. 配置驱动与数据资产

**做了什么：**  
把武器、敌人、波次、流程文案、UI文本、掉落参数，总之我尽可能的把这些内容做成了Data里面的可配置资产。

**怎么设计：**  
`ScriptableObject` + `Provider`模式（如`FlowConfigProvider`、`RunCatalog`、`WaveConfig`）。

**链路：**  
Boot场景来安装全局配置 -> 各系统按Provider读取配置执行。



最初的设计目标是按照进阶验收标准做的，不过要想真正的实现更好的表现这方面得花点时间跟美术资源，比如场景啊动画啊斗智斗勇（找资源），所以还是把重心放在代码架构这些地方了。虽说如此基础的验收标准应该是完全能达到的。
虽然实际的游玩内容不是很多，玩法其实也很单一，看着也很粗糙...
不过作业的主要目的就是熟悉流程和编辑器吧，我觉得这个部分还是很有收获的。
